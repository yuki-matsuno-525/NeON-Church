# E-1：検索の作り直し

メモ「検索対象が福音書のみ？検索はもっとちゃんとやりたい」への対応プラン。
**大改修ではなく、段階ごとに独立して push・ロールバックできる**単位に割る。
各段階でテストを書く（backend: pytest / frontend: npm test）。

## 背景（調査で判明した事実）

現状の検索は `GET /api/search/?q=`（`backend/bible/views.py` `SearchView`）。

- **対象は「福音書のみ」ではない**。コードは `Verse.text__icontains=q` で**全書・全訳**を検索する
  （ローカルで「神」= 4,324 節ヒット、全 71,532 節）。→ **本番で福音書しか出ないなら、それは
  本番DBに全書がインポートされていない**（ibibles の投入が未実行）可能性が高い。段階0で切り分ける。
- **訳の重複**：訳スコープが無く、同じ箇所が訳ごとに別結果で出る（口語訳＋文語訳＝日本語2件、
  KJV／TR＝英語・ギリシャ語…）。節は `[:30]` で頭打ちのため、**重複で枠が埋まり件数が水増し**される。
- **言語を考慮しない**：日本語クエリでも英語・ギリシャ語の節・書名が混ざる（逆も同じ）。
- **並び順に関連度が無い**：`book.order` 昇順のみ。完全一致・語境界・出現頻度などの重み付けが無い。
- **件数が固定・ページング無し**：節 `[:30]` / コメント `[:20]` / 書は無制限。「もっと見る」が無い。
- **性能**：`icontains` は `LIKE '%q%'` で**索引が効かず全表走査**。71,532 節×コメントで、
  データが増えるほど遅くなる（現状は許容範囲だが将来のボトルネック）。
- **書名検索**：`Book.name__icontains` で**投稿時訳の書名**しか当たらない（口語訳名・英名など訳ごと）。
- フロント（`frontend/src/app/search/page.tsx`）：`q` 1本のみ送信、絞り込み無し、
  ハイライトはクライアント側、結果は「書／節／コメント」の3セクション。

返却フィールド：
- 節 `VerseSearchSerializer` = `id/number/text/chapter_number/chapter_id/book_name/book_id`（slug・canonical 無し）
- コメント `CommentSearchSerializer` = `id/body/username/created_at/location`

## 原則

1. 抽象化は最小限（初心者が読んで直せることを最優先）。全文検索基盤の大掛かりな導入は段階3まで保留。
2. 「正確さ（重複・言語）」→「絞り込み・件数」→「性能」→「磨き込み」の順。前段が後段の前提。
3. 各段階は独立して push・ロールバック可能。まとめて作らない。
4. 箇所の同一性は既存どおり訳非依存の `canonical_book`/章/節で扱う（新しい箇所概念を作らない）。
5. まず**データ（段階0）を確定**してから中身をいじる。コードを直しても本番にデータが無ければ意味がない。

---

## 段階0：「福音書のみ」の切り分け（コード変更なし）

**目的**：ユーザー体感「検索が福音書だけ」がコードの問題か本番データの問題かを確定する。

**やること**：
1. 本番（Render）シェルで書数・訳を確認：
   `Book.objects.values_list('translation').annotate(n=Count('id'))` と `Verse.objects.count()`。
2. 福音書以外（例：詩篇・創世記）が本番に無ければ、`backend/bible/seed/ibibles/README.md` の手順で
   `import_ibibles` を全訳ぶん投入（冪等）。→ これだけで「福音書のみ」が解消する見込み。

**完了条件**：本番の全書・全訳の有無が判明し、未投入なら投入して検索に全書が出ること。
**注意**：本番データ操作は不可逆に近い。実行前に現状を控える。

---

## 段階1：訳の重複解消＋言語スコープ（正確さ）

**目的**：同じ箇所が訳ごとに何件も出るのをやめ、UI 言語に合った結果を返す。

**方針（最小変更）**：
- フロントは検索時に **UI 言語（lang）** を送る（`searchBible(q, lang)` → `?q=&lang=ja|en`）。
- backend は節検索を**その言語の訳に絞る**（例：ja → 口語訳・文語訳、en → KJV、原文は別扱い/後回し）。
  訳→言語の対応は既存 `TRANSLATION_LANG`（フロント）と揃える形で backend にも最小の対応表を置く。
- さらに**同一箇所（canonical_book/章/節）で1件に集約**し、代表訳（口語訳優先）を返す。
  結果に `book_slug` を含め、フロントのリンク生成を安定させる（現状は book_name→slug 逆引き）。

**変更対象**：`SearchView`、`VerseSearchSerializer`（`book_slug` 追加）、`frontend .../search/page.tsx`、`api.ts`。
**テスト**：日本語クエリで英語訳が出ない／同一箇所が1件に集約される／`book_slug` が入る。
**完了条件**：重複が消え、言語に合った節が返る。

---

## 段階2：絞り込み（フィルタ）とページング（件数）

**目的**：件数を必要なだけ見られるようにし、対象を絞れるようにする。

**やること**：
- **種別フィルタ**：節／コメント／書（チップ）。既存の3セクションを、選択で切替できるように。
- **書・ジャンル絞り込み**（任意）：`/read` と同じジャンル軸、または書 slug で節検索を絞る。
- **ページング／もっと見る**：固定 `[:30]/[:20]` をやめ、`page`/`page_size`（既存 `StandardPageNumberPagination` を流用）。
- 空状態・ヒット件数表示は現状の文言を流用（新規テキストは足さない）。

**変更対象**：`SearchView`（クエリパラメータ追加）、`frontend .../search/page.tsx`、`api.ts`。
**テスト**：種別・書での絞り込み、ページングの境界。
**完了条件**：フィルタとページングが効く。

---

## 段階3：性能（pg_trgm 索引で icontains を高速化）

**目的**：`LIKE '%q%'` の全表走査を索引化し、データ増加に耐える。

**やること**：
- Postgres 拡張 `pg_trgm` を有効化するマイグレーション（`CREATE EXTENSION IF NOT EXISTS pg_trgm`）。
- `Verse.text` と `Comment.body` に **GIN trigram 索引**（`GinIndex(..., opclasses=["gin_trgm_ops"])`）。
- 挙動は変えず（`icontains` のまま）索引だけで高速化。必要なら `TrigramSimilarity` で関連度並びに拡張。

**変更対象**：`bible`/`comments` のマイグレーション、モデルの `Meta.indexes`。
**テスト**：既存検索テストが緑のまま（挙動不変）。EXPLAIN で索引使用を確認（手動）。
**完了条件**：索引が効き、大きめデータでも検索が実用速度。日本語部分一致も trigram で当たる。
**メモ**：Postgres の全文検索（`to_tsvector`）は日本語トークナイズが弱いので採らない。substring 一致には trigram が適。

---

## 段階4：磨き込み（任意・後回し可）

- 書名検索を**全訳・全言語名**（canonical 単位）で当てる。
- 関連度順（完全一致・語頭一致を優先）。
- ヘッダ等どこからでも検索できる導線（既にあれば省略）。
- 翻訳プロジェクトを検索対象に含めるか（C-3 の設計と整合）。

---

## 進め方メモ

- まず**段階0**で本番データを確定（ここで「福音書のみ」が直る可能性が高い）。
- コード変更は**段階1（正確さ）から**。段階2・3は独立して足せる。
- 段階3のマイグレーション（拡張・GIN 索引）は本番でも流すので、投入手順を PR に明記する。
