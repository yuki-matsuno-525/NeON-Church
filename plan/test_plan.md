## テスト設計

MVPでは、すべてを網羅的にテストするのではなく、サービスの中核機能と壊れると影響が大きい部分を優先してテストする。

NeON-Churchにおける重要機能は以下である。

- ユーザー認証
- 聖書本文の取得
- コメント投稿・返信・削除
- コメントの論理削除
- upvote
- ブックマーク
- 通知
- 本文データimport
- CSRF対策
- 連続投稿制限
- ログの機密情報マスキング

テストは大きく以下の3種類に分ける。

1. バックエンドテスト
2. フロントエンドテスト
3. E2Eテスト

MVPではバックエンドテストを最も重視する。  
理由は、認証、コメント、upvote、ブックマーク、通知、論理削除など、サービスの重要なロジックがバックエンドに集中しているため。

---

### バックエンドテスト

バックエンドテストでは、主に以下を確認する。

- モデルの制約が正しく動くこと
- APIが期待通りに動くこと
- 認証・権限が正しく制御されること
- データが意図せず消えたり重複したりしないこと
- セキュリティ上必要な制御が機能していること

使用する主なツールは以下とする。

- pytest
- pytest-django
- Django REST framework APIClient

#### 認証テスト

JWT + HTTP-only Cookie を採用するため、認証まわりは重点的にテストする。

確認する内容：

- ユーザー登録できる
- username が重複している場合は登録できない
- メールアドレスとパスワードでログインできる
- ログイン時に access_token / refresh_token が HTTP-only Cookie にセットされる
- ログアウト時に Cookie が削除される
- ログアウト時に refresh token が失効する
- ログアウト後は認証が必要なAPIにアクセスできない
- refresh token rotation が機能する
- 古い refresh token は再利用できない
- 複数端末ログインが可能である

#### 聖書本文テスト

書、章、節のデータが正しく取得できることを確認する。

確認する内容：

- 書一覧を取得できる
- 章一覧を取得できる
- 章に含まれる節一覧を取得できる
- 節が正しい順番で返る
- 存在しない書・章・節を指定した場合は404になる
- 匿名ユーザーでも本文を読める

#### 本文importテスト

聖書本文はソースごとにimport処理を作成し、manage.pyから呼び出してDBに格納する。  
この処理は何度も実行される可能性があるため、重複登録を防ぐテストを行う。

確認する内容：

- importコマンドで4福音書のデータを投入できる
- importコマンドを再実行してもデータが重複しない
- 書、章、節の順番が正しく保存される
- 不正な形式のデータではimportに失敗する
- 空データではimportに失敗する

#### コメントテスト

コメントはサービスの中心機能であるため、重点的にテストする。

確認する内容：

- ログインユーザーはコメントを投稿できる
- 匿名ユーザーはコメントを投稿できない
- 書・章・節にコメントを紐づけられる
- コメントに返信できる
- 親コメントIDによってツリー構造を表現できる
- 投稿後にコメントを編集できない
- 自分のコメントは削除できる
- 他人のコメントは削除できない
- コメントは物理削除されない
- コメント削除時は論理削除になる
- 論理削除後もDB上にレコードが残る
- 論理削除後の本文は「このコメントは削除されました」と表示される
- 親コメントが削除されても子コメントは表示される
- コメントを新しい順で取得できる
- コメントをvote数の多い順で取得できる

#### upvoteテスト

コメントにはupvoteのみを付けられる。  
同一ユーザーによる二重投票を防ぐことを重視する。

確認する内容：

- ログインユーザーはコメントにupvoteできる
- 匿名ユーザーはupvoteできない
- 同じユーザーが同じコメントに2回upvoteできない
- upvote数が正しく増える
- 削除済みコメントにはupvoteできない
- vote数順でコメントを並び替えできる

#### ブックマークテスト

ブックマークは認証済みユーザーのみ利用できる。  
対象は、書、章、節、コメントとする。

確認する内容：

- ログインユーザーはブックマークできる
- 匿名ユーザーはブックマークできない
- 書をブックマークできる
- 章をブックマークできる
- 節をブックマークできる
- コメントをブックマークできる
- 同じ対象を二重にブックマークできない
- 自分のブックマーク一覧を取得できる
- 他人のブックマークは取得できない
- 他人のブックマークは削除できない

#### 通知テスト

コメントに返信があった場合、対象ユーザーに通知を作成する。

確認する内容：

- 自分のコメントに返信された場合、通知が作成される
- 自分で自分に返信した場合、通知は作成されない
- 通知は未読状態で作成される
- 自分の通知一覧を取得できる
- 通知を既読にできる
- 他人の通知は取得できない
- 他人の通知を既読にできない

#### 通報・モデレーションテスト

不適切なコメントを報告できるようにする。

確認する内容：

- ログインユーザーはコメントを報告できる
- 匿名ユーザーはコメントを報告できない
- 同じユーザーが同じコメントを複数回報告できない
- 管理者は報告されたコメントを確認できる
- 管理者はコメントを非表示にできる

#### セキュリティテスト

認証、CSRF、投稿制限、ログ保護を確認する。

確認する内容：

- 書き込み系APIはCSRFトークンなしでは失敗する
- CSRFトークンありの場合、書き込み系APIが成功する
- GET系APIはCSRFなしでも実行できる
- Cookieに HttpOnly が付与されている
- 本番環境ではCookieに Secure が付与されている
- Cookieに SameSite が設定されている
- 同一IPからの連続投稿制限が機能する
- 停止されたユーザーはコメントを投稿できない
- ログにコメント本文、パスワード、認証トークン、Cookie、メールアドレスを出力しない
- request_id によってAPIアクセスログとSentryのエラーを紐づけられる

### フロントエンドテスト

フロントエンドテストでは、画面の細かい見た目ではなく、主要な表示と操作を確認する。

使用する主なツールは以下とする。

- Vitest
- React Testing Library

確認する内容：

- ログインフォームが表示される
- ログイン失敗時にエラーが表示される
- ログイン済みの場合、ユーザー情報が表示される
- 書一覧が表示される
- 章一覧が表示される
- 章本文が表示される
- 節番号が表示される
- コメント一覧が表示される
- 未ログイン時はコメント投稿フォームではなくログイン誘導が表示される
- ログイン時はコメント投稿フォームが表示される
- 返信フォームを開ける
- 削除済みコメントは「このコメントは削除されました」と表示される
- vote数順と新しい順を切り替えられる
- ブックマークボタンが表示される
- 通知一覧が表示される
- 未読通知が分かる

デザインの余白、色、細かいレイアウト崩れはMVPではテスト対象外とする。  
まずは、ユーザーが主要機能を利用できるかを優先して確認する。

### E2Eテスト

E2Eテストでは、実際のユーザー操作に近い形で、主要な導線が壊れていないことを確認する。

使用する主なツールは以下とする。

- Playwright

E2Eテストは実行に時間がかかり、壊れやすいため、MVPでは本数を絞る。

#### E2E 1: 聖書本文を読む

確認する流れ：

1. トップページを開く
2. 書一覧からマタイを選ぶ
3. 章一覧から5章を選ぶ
4. 本文が表示される
5. 節番号が表示される
6. 特定の節アンカーにアクセスできる

#### E2E 2: 登録・ログイン

確認する流れ：

1. ユーザー登録する
2. ログアウトする
3. ログインする
4. ログイン済み状態になっていることを確認する

#### E2E 3: コメント投稿・返信・削除

確認する流れ：

1. ログインする
2. 任意の章ページを開く
3. 節にコメントを投稿する
4. 投稿したコメントが表示される
5. コメントに返信する
6. 返信がツリー表示される
7. コメントを削除する
8. 「このコメントは削除されました」と表示される

#### E2E 4: upvote

確認する流れ：

1. ログインする
2. コメントにupvoteする
3. vote数が増える
4. 同じコメントに二重upvoteできない
5. vote数順の並び替えで反映される

#### E2E 5: ブックマーク

確認する流れ：

1. ログインする
2. 節またはコメントをブックマークする
3. ブックマーク一覧に表示される
4. ブックマークを解除できる

MVP初期では、最低限以下の3本を優先する。

- 聖書本文を読む
- 登録・ログイン
- コメント投稿・返信・削除

### MVPで必須のテスト

MVP公開前に、最低限以下のテストが通っている状態を目標とする。

#### 認証

- ユーザー登録できる
- username重複時は登録できない
- ログインできる
- ログイン時にCookieがセットされる
- ログアウトできる
- ログアウト後は認証が必要なAPIにアクセスできない
- refresh token rotationが機能する

#### 聖書本文

- 書一覧を取得できる
- 章一覧を取得できる
- 節一覧を順番通り取得できる
- 本文importを再実行しても重複しない

#### コメント

- ログインユーザーはコメントできる
- 匿名ユーザーはコメントできない
- コメントに返信できる
- コメントは編集できない
- 自分のコメントは削除できる
- 他人のコメントは削除できない
- 削除後もDBにレコードが残る
- 削除後の本文が「このコメントは削除されました」になる

#### upvote

- ログインユーザーはupvoteできる
- 匿名ユーザーはupvoteできない
- 同じコメントに2回upvoteできない
- vote数順で並び替えできる

#### ブックマーク

- ログインユーザーはブックマークできる
- 匿名ユーザーはブックマークできない
- 同じ対象を2回ブックマークできない
- 自分のブックマークだけ取得できる

#### 通知

- 返信されたら通知が作成される
- 自分で自分に返信しても通知されない
- 通知を既読にできる
- 他人の通知は見られない

#### セキュリティ

- 書き込み系APIはCSRFが必要
- 同一IPからの連続投稿制限が機能する
- ログにCookie、トークン、パスワード、メールアドレス、コメント本文が出力されない

### CIでの自動テスト

GitHub Actionsを使い、Pull Request作成時とmainブランチへのpush時にテストを自動実行する。

実行する内容：

#### バックエンド

- Djangoの設定チェック
- マイグレーション漏れチェック
- pytest
- OpenAPI schema生成チェック

実行例：

```bash
python manage.py check
python manage.py makemigrations --check --dry-run
pytest
python manage.py spectacular --file schema.yaml --validate
