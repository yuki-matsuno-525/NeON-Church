# C-3：翻訳プロジェクトの作り込み（A-1・C-2 を統合）

メモ「翻訳プロジェクトの使用感が全体的に微妙。もっとちゃんと作りたい。安っぽい。
レビュー対象からユニットに飛べるように」＋ A-1（Q&A投稿できない）＋ C-2（Q&Aで翻訳PJを選ぶ）を束ねる。
**方針決定が要る論点が多い**ので、まず設計を固めてから段階実装する。

## 背景（調査で判明した事実）

- **コメント系統が2つある**（A-1 の原因）：
  1. プロジェクト詳細/作業ページ `/translations/[id]` … `TranslationComment`（`postUnitComment`）。
     **body のみ・Q&A非対応**の「翻訳作業のレビュー用」コメント。ユニット(節)に紐づく。
  2. 読書ページ `/translations/[id]/read/[chapter]` … 通常の `Comment`（`translation_project` FK 付き）。
     **Q&A対応**。HTTP で投稿成功を確認済み（壊れてはいない）。
  → ユーザーが①でQ&Aを投げようとして「できない」と感じたのが A-1。
- **モデル**：
  - `TranslationProject`（source_book / target_language / status=draft/active/published / メンバー）。
  - `TranslationUnit`（節単位。status: todo→in_progress→review→done、assigned_to、body）。
  - `TranslationComment`（unit 紐づき・body のみ）。
  - `Comment`（`translation_project` 付きで Q&A 可。読書ページ用）。
- **Q&A一覧・投稿は聖書本体のみ**（`translation_project__isnull=True` で絞っている）。
  翻訳PJ の Q&A は `/qa` に出ない（C-2）。
- 絵文字（📖🌐）は C-1 で除去済み。

## 決めるべき論点（要ユーザー判断）

1. **コメント系統を統一するか**：`TranslationComment` を廃し `Comment`（translation_project 付き）に一本化するか、
   役割を分けて残すか（作業レビュー＝ユニット / 議論＝Q&A）。統一は移行コスト大。
2. **Q&A に翻訳PJ を含めるか**（C-2）：`/qa` のフィルタ・投稿先に翻訳PJ を出すか。出すなら一覧の絞りと投稿導線をどう見せるか。
3. **「安っぽさ」の具体**：どの画面のどこが安っぽいか（G-1 のデザインレビューと連動）。
4. **レビュー導線**：レビュー対象（review 状態のユニット）から対象箇所（ユニット/原文）へ飛べる導線をどこに置くか。

## 段階案（論点決定後に着手）

### 段階0：現状のUX棚卸し（コード変更なし）
- 作業ページ／読書ページ／一覧の3画面のスクショを撮り、G-1 と合わせて「安っぽい」箇所を具体化。
- 上記4論点をユーザーと確定する。

### 段階1：レビュー導線（小さく効く）
- 目的：メモ「レビュー対象からユニットに飛べるように」。
- レビュー中(review)ユニットの一覧から、その箇所（読書ページの該当章・節／作業ページの該当ユニット）へリンク。
- 変更は表示・リンクのみ。モデル変更なし。**単独で出せる**。

### 段階2：Q&A と翻訳PJ の接続（C-2、論点2の決定に従う）
- `/qa` のフィルタ・投稿フォームに翻訳PJ を選べるようにする。
- backend の Q&A 一覧が `translation_project__isnull=True` で固定なのを、PJ 指定時はそのPJのQ&Aを返すよう拡張。
- A-1 は「作業ページのコメント欄からもQ&Aを投げたい」なら、そこを `Comment`(Q&A) 経路に寄せる（論点1に従う）。

### 段階3：コメント系統の整理（論点1の決定に従う・大）
- 統一する場合：`TranslationComment` の役割を `Comment` に寄せ、作業ページを `Comment`（unit/箇所紐づき）で描く。
  データ移行（既存 TranslationComment → Comment）を段階的に。**独立PR・ロールバック可能に割る**。
- 分けて残す場合：2系統の「役割の違い」を UI で明示（作業メモ vs 議論）。

### 段階4：見た目の作り込み（G-1 連動）
- カード・余白・タイポ・状態バッジなどを整える（デザイン方針が出てから）。

## 原則
1. まず論点を確定（段階0）。UI/データ変更は決定後。
2. 効果が大きく低リスクな**段階1（レビュー導線）から**着手できる。
3. コメント統一（段階3）は移行を伴うので独立PRで慎重に。
4. 見た目（段階4）は G-1 の方針とそろえる。

- デザイン面は G-1（`plan/design-review.md` 予定）と連動。
