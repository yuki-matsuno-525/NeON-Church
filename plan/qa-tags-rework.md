# D-2：Q&A タグ項目の再考

メモ「タグ項目再考」。Q&A コメントに付けるタグの種類・粒度を見直す。**要ユーザー判断**（何を残す/足す/削るか）。

## 背景（現状の事実）

- タグは**固定の定義済みのみ**（自由入力なし）。`comments/models.py` `PREDEFINED_TAGS`。
  現在：**感想 / 解説 / 証し / 祈り / 考察**（「質問」は migration 0008 で削除済み）。
- 付与先：Q&A コメント（`Comment.tags` M2M）。投稿フォームで選択、`/qa` でタグ絞り込み。
- 追加・変更は migration（`0007_seed_predefined_tags` と同様の RunPython）で行う。

## 決めるべき論点（要ユーザー判断）

1. **タグ体系の方向性**：
   - (a) 定義済みタグを**磨く**（種類の追加・削除・改名）。今のまま固定運用。
   - (b) **自由入力タグ**を許す（ユーザーが任意に付ける）。表記ゆれ・モデレーションの課題あり。
   - (c) **2軸に分ける**（例：内容種別＝質問/解説/考察 と 目的＝祈り/証し）。
2. **タグの必須/任意**：投稿時にタグ必須にするか任意のままか。
3. **既存データの扱い**：タグを改名/削除する場合、既存コメントのタグをどう移すか。

## 段階案（方向性決定後）

### 段階0：現状棚卸し（コード変更なし）
- 各タグの使用件数を集計（`Tag` ごとの付与数）。使われていないタグ・不足しているタグを可視化。
- 上記論点をユーザーと確定。

### 段階1：定義済みタグの調整（方向性(a)なら）
- `PREDEFINED_TAGS` を更新する migration（追加/改名/削除、既存付与の移行込み）。
- フロントのタグ表示・絞り込みは既存のまま流用（`fetchTags` が返す一覧に追従）。

### 段階2：自由入力/2軸（方向性(b)(c)なら・大）
- (b) 自由入力：Tag の作成経路・正規化・上限・モデレーションを設計。
- (c) 2軸：`Tag` に区分（category）を持たせる or タグ種別テーブルを足す。UI も2軸に。

## 原則
1. まず方向性を1つ選ぶ（段階0）。データ移行を伴う変更は独立 migration で。
2. 自由入力（b）は表記ゆれ・荒れやすさのコストが大きい。まず (a) の磨き込みで足りるか確認する。
3. デザイン（タグの見せ方）は G-1 と連動。
